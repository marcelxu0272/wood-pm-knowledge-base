# 现金流仪表盘需求文档

## 1. 背景与目标

### 1.1 背景
- 2026 年 4 月管理会明确：利润共享兑现前提为 **P>0 且现金流为正**
- 当前经营数据分散在月度经营数据、开票回款、各类付款申请等模块，缺乏统一的现金流视角
- 各板块负责人、项目群负责人、管理层对现金流健康度缺乏实时感知

### 1.2 目标
- 建设覆盖 **板块 → 项目群 → 公司** 三级联动的现金流仪表盘
- 集中展示 WIP、存货、AR（应收账款）、Cash In（回款）、Cash Out（付款）五大核心指标
- 为利润共享兑现、考核判定、资金调配提供数据支撑
- 实现异常预警（超期 AR、负现金流、回款率偏低等）

## 2. 数据源盘点

### 2.1 已有数据源

- **月度经营数据设置**（`03_数据与规则`）
  - 字段：存量合同额、WIP、应收账款、完成合同额、完成开票额、完成回款额
  - 维度：年 × 月 × 板块 × 目标类型（设计及咨询 / EPC / PMC）
  - 机制：每月初人工导入上月数据
  - 性质：存量值（WIP、AR、存量合同额）与流量值（完成合同额、开票额、回款额）并存

- **开票回款（中台）**
  - 开票日期、回款日期统计口径
  - 预警邮件机制（通知开票创建人、板块负责人、沈风华、刘军）

- **付款模块**（分散）
  - 采购付款申请（含支付时间作为成本统计口径）
  - 施工付款申请
  - 一般付款申请
  - 费用报销（外埠差旅、市内出差、探亲路费）
  - 往来款申请

- **组织架构**（`05_组织与业务/组织架构.md`）
  - 五大项目群：SAS810 东部 / SAS820 西部 / SAS830 海外 / SAS840 NM&LS / SAS850 新业务
  - 各板块归属关系明确

- **存货（Inventory）**
  - 现状：**平台暂无数据，财务系统无接口**
  - ~~计算逻辑（理想状态，实际由财务负责）：`存货 = 实际成本 - (平台开票金额 × (1 - 预算利润率))`~~
    - ~~财务设定目标毛利率（如 25%）→ 预测成本率（75%）~~
    - ~~用平台开票金额 × 预测成本率 = 预测成本~~

  - 缺口：平台缺乏存货数据维护入口
  - 方案：Phase 1 暂不展示；Phase 2 在月度经营数据设置中新增"存货"字段，由财务人员每月初随经营数据一并手工导入

- **Cash Out（现金流出）聚合**
  - 现状：付款数据散落在多个模块，无统一出口
  - 缺口：没有按板块 / 项目归集的现金流出总额
  - 建议：Phase 1 先抽取主要付款模块汇总，Phase 2 建立统一付款流水表

- **项目维度明细**
  - 现状：月度数据只到"板块 × 目标类型"
  - 缺口：无法下钻到项目群 / 单体项目
  - 建议：Phase 2 对接财务系统或建立项目级数据维护入口

- **AR 账龄分析**
  - 现状：仅有应收账款总额
  - 缺口：无按账龄（30 / 60 / 90 / 180+ 天）分层
  - 建议：Phase 2 基于开票日期 + 回款日期明细计算

- **现金流预测**
  - 现状：无
  - 缺口：缺乏基于合同里程碑 / 付款计划的前瞻数据
  - 建议：Phase 3 建设

## 3. 三级看板架构

### 3.1 层级关系

```
公司级看板（ELT / Board / 总经理及管理团队）
  │
  └── 下钻 ──→ 项目群级看板（SAS810 ~ SAS850 五大项目群负责人）
                  │
                  └── 下钻 ──→ 板块级看板（各中心 / 板块负责人）
```

### 3.2 各层级覆盖范围

**公司级（SAS000）**
- 覆盖范围：全公司所有板块汇总
- 使用者：Board、总经理及 ELT、SAS180 绩效与财务部
- 核心用途：全局资金健康度监控、共享兑现判定、战略决策

**项目群级（SAS810 ~ SAS850）**
- SAS810 东部项目群：金山中心、沈阳中心、惠湛中心
- SAS820 西部项目群：银川中心、公司级 EPC 项目
- SAS830 海外项目群：COII 板块、模块化板块、供应链板块
- SAS840 NM&LS 项目群：新材料板块、生命科学板块
- SAS850 新业务项目群：PMC 板块、咨询板块、数字技术板块
- 使用者：各项目群负责人
- 核心用途：项目群间资源调配、回款进度对比、异常定位

**板块级（SAS520 ~ SAS720 及各业务板块）**
- 覆盖范围：单体中心或业务板块
- 使用者：板块负责人（如赵宏海、杨科、陈贺明等）
- 核心用途：板块经营自查、回款催办、WIP 积压管理

## 4. 指标定义与计算逻辑

### 4.1 核心五大指标

- **WIP（Work In Progress，在实施项目累计确认产值）**
  - 来源：月度经营数据设置
  - 类型：存量值（累计值）
  - 意义：反映在手项目的交付进度与潜在收入

- **存货（Inventory）**
  - 来源：财务手工导入（Phase 1 暂不展示，Phase 2 随月度经营数据一并导入）
  - 计算公式：`存货 = 实际成本 - (平台开票金额 × (1 - 预算利润率))`
  - 业务含义：没有收入分摊的那部分成本（已发生成本 vs 预测成本的差额）
  - 意义：资金占用情况。存货越高，说明垫付资金越多，现金流越紧
  - **已确认口径**：
    - 收入基数：**平台开票金额**（从开票回款模块获取）
    - 预算利润率：财务设定目标值（如 25%），预测成本率 = 1 - 预算利润率
    - 实际成本范围：与财务结转口径一致（含直接成本 + 管理费分摊，具体由财务定义）

- **AR（Accounts Receivable，应收账款）**
  - 来源：月度经营数据设置 + 开票回款（中台）
  - 类型：存量值（累计值）
  - 计算：已确认收入但尚未收到的款项
  - 意义：资金回收效率，AR 过高意味着收入已确认但现金未回笼

- **Cash In（现金流入 / 回款）**
  - 来源：月度经营数据设置（完成回款额）+ 开票回款（中台）
  - 类型：流量值（当月值）
  - 意义：实际到账资金

- **Cash Out（现金流出 / 付款）**
  - 来源：采购付款 + 施工付款 + 一般付款 + 费用报销 + 往来款（Phase 1 汇总）
  - 类型：流量值（当月值）
  - 意义：实际支出资金

### 4.2 衍生指标

- **净现金流** = Cash In - Cash Out
  - 正数：资金健康
  - 负数：资金紧张，需预警

- **回款率** = 完成回款额 / 完成开票额
  - 反映开票后实际回款的比例
  - 目标值：建议设定基准线（如 ≥85%）

- **DSO（Days Sales Outstanding，应收账款周转天数）**
  - Phase 2 建设，需 AR 账龄明细
  - 反映从开票到回款的平均周期

- **WIP / 存量合同额 比率**
  - 反映在手合同的交付进度
  - 比率过低：交付滞后
  - 比率过高：可能已接近交付尾声，需关注新签订单

- **现金转换周期（CCC）**
  - Phase 2 建设
  - CCC = DSO + DIO（存货周转天数）- DPO（应付账款周转天数）

### 4.3 共享兑现判定指标

结合管理会考核逻辑：

- **利润贡献（P）** = 开票收入（R）- 总成本 - 管理费（OH）
- **兑现前提**：P > 0 **且** 现金流为正（Cash In - Cash Out > 0）
- 看板需展示：各板块是否满足兑现条件（🟢 绿灯 / 🔴 红灯）

## 6. 建设路径

### Phase 1：板块级基础看板（2 ~ 3 周）

**范围**
- 基于现有「月度经营数据设置」+「开票回款（中台）」
- 覆盖板块级 WIP、AR、Cash In、Cash Out（从付款模块汇总）
- 实现板块 → 项目群 → 公司三级汇总联动

**数据接入**
- 复用月度经营数据导入流程
- 从采购付款、施工付款、一般付款、费用报销模块提取 Cash Out 并汇总

**展示**
- 核心指标卡 + 近 6 个月趋势图
- 板块健康度排名
- 回款率计算

**交付物**
- 现金流仪表盘 MVP
- 数据字典（指标定义 + 计算口径）

### Phase 2：项目级下钻、账龄分析与存货接入（1 ~ 2 月）

**新增数据源**
- **存货数据接入**：在月度经营数据设置中新增"存货"字段，由财务人员每月初随经营数据手工导入
- 建立项目级数据维护入口（或对接财务系统）
- 接入开票明细（开票日期 + 回款日期）用于 AR 账龄计算

**新增功能**
- AR 账龄分层（30 / 60 / 90 / 180+ 天）
- 项目级下钻
- DSO 计算
- 大项目付款节点追踪
- 存货指标上线（资金占用监控）

### Phase 3：前瞻性预测与自动化（季度级）

**功能**
- 基于合同里程碑的预期现金流预测
- 预算 vs 实际对比
- 共享兑现自动判定（P > 0 且现金流为正 → 自动绿灯）
- 智能预警推送（企业微信 / 钉钉 / 平台消息）

## 7. 关键业务规则

### 7.1 数据刷新机制

- 月度经营数据：每月初人工导入（沿用现有机制）
- Cash Out 汇总：建议 Phase 1 采用定时任务每日/每周汇总付款模块数据
- 实时性要求：Cash In/Cash Out 建议 T+1 更新，WIP/AR 可月度更新

### 7.2 权限控制

- 公司级：ELT、SAS180 绩效与财务部、总经理可查看全部
- 项目群级：项目群负责人可查看所属板块 + 项目群汇总
- 板块级：板块负责人仅可查看本板块数据
- SAS180 绩效与财务部：全量数据 + 导出权限

### 7.3 预警阈值（建议值，待业务确认）

- 净现金流连续 2 个月为负 → 🟡 预警
- 净现金流连续 3 个月为负 → 🔴 告警
- AR 超 90 天占比 >20% → 🟡 预警
- 回款率 <70% → 🟡 预警
- 回款率 <50% → 🔴 告警

## 8. 风险与注意事项

- **数据口径一致性**：Cash Out 汇总需明确是否包含管理费、报销、往来款，Phase 1 建议先明确范围
- **存货数据人工导入依赖**：财务系统无接口，存货数据依赖财务人员每月手工导入，存在延迟和人为差错风险；需确保导入流程规范并与现有月度经营数据导入合并
- **开票金额 vs 财务收入差异**：存货计算采用平台开票金额，与财务按完工百分比确认的收入可能存在时间差，需业务侧理解并接受该差异
- **人工导入依赖**：月度经营数据依赖人工导入，存在延迟和差错风险，长期应考虑系统自动对接
- **历史数据缺失**：如果历史付款数据不完整，Phase 1 的 Cash Out 趋势可能从当前月开始
- **共享兑现逻辑**：看板仅做数据展示与判定辅助，最终兑现决策仍以管理会决议为准
