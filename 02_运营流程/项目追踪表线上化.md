## 一、项目概述

### 1.1 目标
- 将金山中心（S520）及各执行中心的项目执行跟踪 Excel 表线上化
- 实现多角色协作填报、审批流程、数据变更追踪、自动化报表
- 覆盖公司**所有已签署合同的项目** + **提前申报产值的项目**

### 1.2 原始数据
- **源文件**：`S520_金山中心_项目执行跟踪详细数据2026年05月.xlsx`
- **字段**：83 列，11 个功能分区，约 1220 条项目记录
- **覆盖中心**：所有项目执行中心（参考 PM 知识库 `05_组织与业务`）

### 1.3 项目进度

| 项目 | 负责人 | 当前状态 | 最新进展 | 风险提示 |
|---|---|---|---|---|
| 项目追踪表线上化 | Ethan, 比比 | 前端原型开发中 | 填报页（Luckysheet）、分板块审批流、PM 提交/板块接收、存量指标 R/S 预警、平台数据同步框架、项目详情 Drawer、系统管理员底栏、Diff 高亮与批注等已实现 | 财务系统接口对接复杂度、多板块数据权限隔离、大数据量下 Luckysheet 性能 |

## 二、核心架构

### 2.1 六大角色

| 角色 | 定位 | 编辑权限 | 审批权限 |
|---|---|---|---|
| **系统管理员** | 系统运维 | ✅ 全公司 | △ 归档确认 |
| **财务审核** | 数据校验 | ❌ 只读 | ❌ 无 |
| **板块管理员** | 数据管理 | ✅ 板块内 | ❌ 无 |
| **项目经理** | 数据填报 | ✅ 本人项目 | ❌ 无 |
| **板块总监** | 审批决策 | △ 待办节点 | ✅ 初审 |
| **项目群群主** | 审批决策 | △ 待办节点 | ✅ 复审 |

### 2.2 审批流程

```
板块管理员提交审批
  → 板块总监审批（Approve1）
    → 项目群群主审批（Approve2）
      → 系统管理员归档（J 版，全公司一张表）
```

- 每次状态变更自动生成**月度快照版本**
- J 版归档与各板块 Approve2 进度**解耦**

### 2.3 填报周期与锁定

| 时间 | 事件 |
|---|---|
| 1-3 日 | 财务核查提醒（次月，不改变锁定规则） |
| 19 日 | 填报提醒 |
| 25 日 | 月度锁定，数据冻结 |
| 次月 9 日 | 常规解禁 |

## 三、521 会议讨论要点

> 参会人员：李总（李永辉）、彭军（产品经理）、许月娟、徐亦盛（Ethan）
> 背景：线上化需解决**历史线下表格数据**与**线上持续新增数据**的融合问题

### 3.1 核心挑战：历史数据与线上数据融合

**问题描述：**
- 历史数据存在于线下 Excel 表格中（S520 金山中心项目执行跟踪表等）
- 线上化后，每月持续新增数据（新签项目、月度填报、审批快照）
- 需要建立从线下到线上的**平滑迁移机制**和**数据初始化流程**

**涉及场景：**
1. **初始导入**：将历史 Excel 数据一次性导入系统作为基准
2. **前期项目转正**：未签署项目（带尾巴的临时项目号）签署后自动切换为正式项目号
3. **跨年数据衔接**：明年年初的初始化逻辑与当年不同

### 3.2 审批流程设计

#### 3.2.1 审批端必须使用 PC 端

**李总明确要求：**

> "不要给他做企业微信端，这数据量太大了。"
> "越给他们便利，他们越肆无忌惮。"

| 决策 | 说明 |
| :-- | :-- |
| **填报 → 网页端** | Luckysheet 类 Excel 界面，数据量大，必须 PC |
| **审批 → 网页端（PC）** | 板块总监、项目群群主审批必须在 PC 端完成 |
| **审批 → 不做企业微信端** | 数据量太大，手机端看不清，容易"瞎批" |
| **理由** | 以前拿一摞纸质文件还会看一眼，现在电子签字越便利越不认真看 |

#### 3.2.2 退回机制简化

| 设计 | 说明 |
| :-- | :-- |
| **退回不强制写 comments** | 很多东西审批人说不清，强制写反而敷衍（写几个字就完了） |
| **推荐线下沟通** | 审批人发现问题 → 直接拿电脑去找填报人说"你这个填错了，改一下" |
| **效率优先** | 直接沟通比系统里写 comments 更高效 |

**示例场景：**
> 让张宏海在系统里写拒绝理由，面对金山那么一大张表，他可能就写几个字就算了。不如直接去找填报人当面说。

#### 3.2.3 版本冲突风险

**彭军 & 徐亦盛关注：**
- 如果板块总监和填报人**同时能改**，可能出现版本冲突
- 审批人审批时看到的是旧版本，填报人在审批过程中又修改了数据

**当前设计：**
- 板块总监/项目群群主在待办节点可编辑与板块管理员相同的填报列
- 审批人上传报告后仍可继续修改
- 需要后续考虑**锁定机制**或**版本快照**在审批节点的保护

### 3.3 报告生成需求

**李总提出：**

> "这个项目是不是还要生成报告才能发给所有的人？"
> "有没有可能提供一个模板工具？"

**需求：**
- 审批完成后需要生成报告发给所有人
- 需要一个**报告模板工具**，让用户按照模板自动生成报告
- 报告内容基于审批通过的最终版本数据

### 3.4 数据初始化逻辑

#### 3.4.1 初始化分为两部分

| 部分 | 说明 |
| :-- | :-- |
| **初始数据** | 历史 Excel 中的存量项目数据一次性导入 |
| **新增项目** | 后续通过新签合同或提前申报产值持续新增 |

#### 3.4.2 新增项目来源

| 来源 | 说明 |
| :-- | :-- |
| **新签合同** | CRB3 审批通过后正式签署的项目 |
| **提前申报产值** | 未签署但需要提前申报产值的前期项目 |

#### 3.4.3 前期项目转正逻辑

**问题：** 提前申报产值的项目变成实际项目后，项目号如何处理？

**讨论结论：**

| 场景 | 处理方式 |
| :-- | :-- |
| 前期项目（未签署） | 项目号**不带尾巴**，通过"已签署/未签署"**标签**区分状态 |
| 转为正式项目 | 项目号不变，状态标签从"未签署"切换为"已签署" |
| 老项目未签署 | 也不带尾巴，通过标签区分 |
| 前期项目最终未签 | **年底清理**，从系统中移除 |
| 跨年处理 | 明年初始化逻辑与当年不同，需单独处理 |

**许月娟确认：**
> "这个表里面最终都不带尾巴。因为你去检索项目的时候，你知道特别是未签，有时候你去搜那个东西它会有问题，所以说它这只是里边的状态，写为是未签署。"

#### 3.4.4 明年初始化差异

- 当年未签署且最终没签的项目 → 年底清理
- 跨年后初始化时，这些项目不应出现在新一年的数据中
- 需要考虑**跨年项目**的延续逻辑（已签署的旧项目继续出现在新年数据中）

### 3.5 财务调整界面需求

**许月娟的操作方式：**
> "我肯定是导出来，我不会在线上导出来，给我看看改了哪些东西，然后完了就单个的去调整。"

**李总指出：**
> "这就是说你要他得要有这个开发给你这个界面。"

**需求确认（彭军）：**
> "这个界面肯定是需要有的呀。"

**具体要求：**
- 所有板块总监审批完成后，数据流转到许月娟（财务/系统管理）
- 许月娟需要**在线调整界面**，不能只靠导出 Excel 再手动调整
- 需要能查看**改了哪些东西**（变更 diff）
- 支持**单个项目逐个调整**

### 3.6 填报体验优化

**已实现/讨论的功能：**

| 功能 | 说明 | 状态 |
| :-- | :-- | :-- |
| 快速填报模式 | 点击后所有可填报的指标放前面 | 已实现 |
| 表单形式展现 | 所有指标以表单形式展示 | 已实现 |
| 项目经理目标 | 部分项目可以不填 | 已确认可不填 |
| 报告月当月完成额清零 | 每月初始化时自动重置为 0 | 已定义 |
| 开票/回款由财务系统同步 | 报告月当月实际值，只读 | 已定义 |
| 历史数据只读 | 报告月之前的数据不可修改 | 已定义 |
| 未来预测可编辑 | 报告月之后的预测值可编辑 | 已定义 |

### 3.7 板块管理员 vs 板块总监

**讨论发现：**
- 正常情况下板块管理员和板块总监**可能是同一个人**
- 但大板块（如新材料）会有独立的板块管理员（如钟晓玲管理，陈康填报）
- 小板块（如供应链）可能各人自己管自己

**对权限设计的影响：**
- 板块总监和板块管理员的权限需要**独立设计**，不能假设是同一人
- 板块总监的审批权限和板块管理员的编辑权限需要**分离**

## 四、数据源规则

| 数据类型 | 数据源 | 说明 |
|---|---|---|
| **项目基本信息** | CRB（合同系统） | 作为基础只读数据 |
| **总合同额** | CRB | 核心真理值，前端不可编辑 |
| **未签署项目** | CRB1 | CRB3 审批通过前取 CRB1 |
| **已签署项目** | CRB3 | 签署后自动切换取 CRB3 |
| **项目时间** | CRB | 严禁手动修改 |
| **回款/现金流** | 财务系统 | 独立取数，不依赖开票 |
| **开票金额** | 财务系统（当月）/ 手工预测（未来月） | 当月实际值只读，未来月可编辑 |
| **存量开票额（R）** | 系统计算 | `P − AC`；小于 0 为预警 |
| **存量合同额（S）** | 系统计算 | `P − U`；不得为负 |

## 五、项目准入规则

### 5.1 正常准入
所有**已签署合同**的项目自动进入追踪表。

### 5.2 提前申报产值（未签署项目）

1. **发起**：针对尚未签署合同但已开展工作需申报产值的项目，由项目经理发起"提前申报产值"流程
2. **审批**：审批通过后，该项目**允许进入**当期追踪表，此时"合同是否签署"字段显示为"未签署"
3. **状态联动**：若后续该项目在 CRB 中完成签署（状态变更为"已确认"），系统自动将追踪表中该项目的状态更新为"已签署"
4. **数据隔离**：未签署且未通过提前申报审批的项目，**不可见**于追踪表中
